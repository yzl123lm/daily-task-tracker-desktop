1. 修订摘要与结论
Cursor 原方案的方向正确：新增 LLM Agent、工具注册表、真实补丁、结构扫描、测试构建与修复闭环。但原方案在工程约束上仍偏粗，尤其是补丁协议、写入边界、LLM 工具兼容、并发取消、staged patch 持久化和测试验收不足。
最终结论
建议先通过本修订方案完成 P0-0 协议与安全底座，再进入只读 Agent Loop、真实补丁 staging、审批写入链路、验证修复闭环。不要让 Cursor 直接按原方案编码。

4.2 统一任务状态机
CREATED → ANALYZING → SCANNING → PLANNING
→ WAITING_APPROVAL → PATCH_READY → APPLYING
→ TESTING → FIXING → REVIEWING → COMPLETED
↘ FAILED / CANCELED / ARCHIVED

后端 REVIEWING 与前端 WAITING_APPROVAL 需要统一映射，避免筛选和状态 pill 不一致。
每次状态切换必须写入 Timeline，包含时间、agentRunId、触发来源和摘要。
FIXING 只在用户授权验证且验证失败后进入，最多 3 轮。

6. PatchEdit 与 StagedPatch 设计
6.1 PatchEdit 协议
{
  "file": "app/workbench/projectArea.js",
  "operation": "replace_range",
  "startAnchor": "function bindConversationClick",
  "endAnchor": "function renderConversationList",
  "oldText": "...",
  "newText": "...",
  "reason": "修复会话点击后未切换 activeConversationId",
  "confidence": 0.82
}

operation
用途
关键校验
replace
替换唯一 oldText
oldText 必须唯一匹配
replace_range
基于起止 anchor 替换区间
anchor 必须稳定且唯一
insert_before
在 anchor 前插入
anchor 必须唯一
insert_after
在 anchor 后插入
anchor 必须唯一
delete
删除指定片段
高风险审批；必须可恢复
create_file
新增文件
不得覆盖已有文件，除非强审批
append_file
追加到文件末尾
限制文件类型和大小
full_content
整文件替换
仅小文件或强审批；大文件禁止

6.2 Patch 校验规则
file 必须通过 assertUnderRoot，禁止路径穿越。
修改前必须 read_file；不能对未读文件生成补丁。
oldText 多处匹配必须失败，不能猜测修改位置。
oldText 不匹配必须失败，让 LLM 重新读取上下文缩小范围。
所有 PatchEdit 必须先生成 unified diff，再进入 DiffReviewPanel。
LLM 不允许调用 writeProjectFile 或 controlledDevService 写入。
6.3 StagedPatch 持久化
{
  "patchId": "patch_xxx",
  "projectId": "proj_xxx",
  "taskId": "task_xxx",
  "agentRunId": "run_xxx",
  "mode": "PATCH_PROPOSE | VERIFY_FIX",
  "files": [],
  "edits": [],
  "diff": "",
  "status": "STAGED | ACCEPTED | REJECTED | APPLIED | FAILED",
  "createdAt": 0,
  "updatedAt": 0
}


关键修正
StagedPatch 必须由主进程 patchStagingService 持久化。渲染层 codeReviewStore 只作为展示状态，不应作为唯一数据源。

7. AgentRun 并发、取消与持久化
同一个 taskId 同一时间只允许一个 active agent run。
每次运行生成 agentRunId，所有工具调用、Patch、Timeline 都必须携带该 ID。
用户切换项目、取消任务或关闭窗口时，当前 run 必须可取消。
runId 不一致的异步结果不得写回 UI，也不得生成 staged patch。
LLM 请求和工具调用都需要 timeout；超时后状态进入 FAILED 或 CANCELED。
应用重启后未完成 run 标记为 CANCELED 或 FAILED，避免悬挂状态。

说明
允许转换
PENDING
等待启动
RUNNING / CANCELED
RUNNING
LLM 或工具执行中
WAITING_APPROVAL / FAILED / CANCELED
WAITING_APPROVAL
等待用户审查 Diff 或授权验证

终态

不要让 Workbench 单独复制一套主对话模型调用逻辑。建议抽取共享 AI 层，由主 AI 对话和 Workbench Agent 共用模型配置、鉴权、错误处理与工具调用适配。

10. 验证与修复闭环
设计原则
自动验证可以被用户授权；自动修复可以生成 staged patch；自动写入仍然禁止。每一轮修复都必须进入 DiffReviewPanel，由用户接受后再写入。

写入成功
→ 用户授权自动验证
→ verificationService 运行 build/test/lint/typecheck
→ parseBuildError 解析 stderr/stdout
→ VERIFY_FIX Agent 读取错误相关文件
→ 生成修复 PatchEdit
→ stage_patch + DiffReviewPanel
→ 用户审批写入
→ 最多 3 轮

命令类型
默认策略
npm run build
可作为验证命令，需用户授权或任务级授权
npm run test
可作为验证命令，需用户授权或任务级授权
npm run lint / typecheck
可作为验证命令，脚本存在才展示
npm run dev
长期运行命令，默认不自动执行
npm install
会改依赖，默认禁止自动执行，需强审批
rm / del / format / shutdown
危险命令，禁止

11. 安全策略与禁止行为
11.1 必须保留的安全能力
路径 assertUnderRoot 与 namespace 隔离。
写入前自动备份。

Git commit 必须用户确认，禁止自动 push。

修改项目目录外文件
路径越权风险
自动 npm install
依赖变更不可控

远程风险不可接受

容易误改代码
fix loop 无限循环

14.2 最终验收标准
Agent 能主动搜索和读取与需求相关的文件。
PLAN_ONLY 只生成基于真实代码的分析方案，不生成可写入 Diff。
PATCH_PROPOSE 能生成真实 unified diff，不再出现“顶部插 PLAN_ONLY 注释”的伪 Diff。
DiffReviewPanel 支持 accept/reject，拒绝不写入。
用户接受后才写入，写入前自动备份。
staged patch 由主进程持久化，刷新页面不丢失。
写入后用户可授权运行 build/test/lint/typecheck。
验证失败时能读取错误并生成二次修复补丁，最多 3 轮。
所有工具调用、审批、验证和状态切换写入 Timeline。
不破坏 Layout v4、现有审批、备份、Shell 白名单、namespace 隔离能力。
