基于《AI 编程错误沉淀与上下文压缩排查报告》的分析与落地设计
生成日期：2026-07-08    ·    适用范围：Workbench / ProjectAgent / Context Compression / Error Lesson
一句话结论
当前系统已经具备“错误被动记录”和“上下文自动压缩”的基础能力，但缺少把错误转化为可复用经验的闭环。建议以 Error Lesson Pipeline 为主线，把错误事件统一采集、结构化、指纹化、双写存储、相似召回、Prompt 注入与验证回写串成闭环，将能力从 L2~L3 提升到 L4+，并为 L5 主动规避打基础。
本方案输出三类内容：第一，针对原报告的能力分析与结构优化建议；第二，面向研发落地的升级架构、数据模型、关键算法与模块清单；第三，P0/P1/P2 实施路线、验收标准、指标体系和风险控制。

4. 升级目标与设计原则
建议把目标从“自动记录错误”升级为“错误经验闭环系统”。目标不是多存日志，而是在未来类似场景中让 Agent 明确知道：以前错在哪里、应该避开什么、验证方式是什么。
目标 1：从 L2~L3 升级到 L4+。系统能自动生成结构化经验规则，并在新任务上下文中稳定注入。
目标 2：为 L5 主动规避打基础。同类错误出现前，Agent 能基于 fingerprint/文件/类别主动提示风险并调整方案。
目标 3：上下文压缩不丢“关键经验”。压缩快照保留 lesson_refs、用户约束、当前错误和已验证决策。
目标 4：让知识可审阅、可回滚、可度量。DB 负责机器检索，Markdown 负责人工审阅，UI 负责治理与恢复。

11. 错误指纹、相似检索与主动规避策略
11.1 指纹生成建议
指纹需要足够稳定，避免路径、临时时间、随机 ID、绝对行号变化导致重复 lesson；同时又要保留类别、文件类型、错误类型等区分度。
清洗 raw stderr/stdout：脱敏 token、绝对路径归一化、移除时间戳、随机 ID、机器路径。
提取 error_type：SyntaxError、AssertionError、TypeError、BuildError、PatchContextMismatch 等。
提取 parser_code：例如 ESLint rule、TypeScript code、pytest assertion pattern、patch failure code。
提取 top_frame 或 primary_issue：优先 file extension + normalized message + nearest function。
拼接 source/category/error_type/parser_code/normalized_message/related_file_ext。
使用 SHA-256 生成 hash，截断为 12~16 位用于展示，完整 hash 用于 DB。
fingerprint_input = [  source,  category,  normalized_error_type,  parser_code || '',  normalizeMessage(primary_message),  normalizeFileKind(primary_file),  top_stack_symbol || ''].join('|')fingerprint = sha256(fingerprint_input).slice(0, 16)
11.2 排序公式建议
Lesson 检索排序不要只看相似度。已验证、复发多、当前文件相关、最近发生的经验应更靠前。
score =  1.00 * exactFingerprintMatch +  0.65 * categoryMatch +  0.55 * fileOverlap +  0.45 * semanticSimilarity +  0.30 * verifiedBoost +  0.20 * log1p(recurrenceCount) +  0.10 * recencyBoost -  0.50 * deprecatedPenalty -  0.30 * failedVerificationPenalty
